# Replication Guide — 可复刻工程实现细节生成器

## 这个 skill 解决什么

用户经常想"借鉴一下某个开源项目某块代码的实现"。但直接读源码代价高：
- 注释里藏着大量"为什么这么写"的故事，但散落在几十个文件里
- 攻击向量、bug 修复、不对称设计这些**最有价值的细节**正是要全文通读才能拼齐的
- 普通的"代码概览"文档抓不住这些 —— 它们关心架构、不关心血泪

本 skill 把这件事系统化：给一个文件夹路径，输出一份 **detail.md**，让用户能**只看这份文档就把功能等价的代码重新写一遍**。

## 输出契约

文件路径：`<入参文件夹>/detail.md`（直接放在被分析的文件夹里）

### 输出行数的硬性要求

行数必须与源码规模成**正比**。下表的上下界是**硬性指标**，低于下限说明细节不够透，必须回去补读代码：

| 源码总行数 | detail.md 目标行数 | 每个源文件的最小出现次数 |
|-----------|-------------------|----------------------|
| < 500 | ≤ 300 | 每个文件至少 1 段具体实现细节 |
| 500 – 2000 | 500 – 800 | 每个文件至少 2 段实现细节（含代码引用） |
| 2000 – 5000 | 800 – 1500 | 每个文件至少 3 段，大文件（>200行）至少 5 段 |
| 5000 – 10000 | 1500 – 2500 | 每个文件至少 4 段，大文件至少 8 段 |
| 10000 – 20000 | 2500 – 4000 | 每个文件至少 5 段，大文件至少 10 段 |
| > 20000 | 4000+，上不封顶 | 每个文件至少 6 段，大文件至少 12 段 |

**每段"实现细节"的定义**：不是一个句子，而是一个完整的"现象 → 攻击/失败场景 → 防御/修复 → [代码定位](path#Lline)"四元组，或者一个"为什么是 N（魔术数）"的完整解释，至少 3–8 行。

结构遵循 [output-template.md](references/output-template.md)，至少覆盖：

1. 整体哲学（这块代码假设什么、解决什么）
2. 文件分工速查表（每个文件多少行、做什么）
3. 接口/门面（对外暴露的形状）
4. 核心数据流（一次典型调用从入口到出口的完整路径）
5. **关键工程细节** —— 这一节是核心，越多越好：
   - 多层防御 / 多模式校验
   - 不动点迭代 / 顺序敏感操作
   - 模式之间的不对称（allow vs deny / 模型视角 vs 用户视角）
   - 解析失败时的 fail-safe 方向
   - 注释里写明的攻击向量与对应防御
   - 一字之差的安全细节（如 `\s+` vs `[ \t]+`）
6. **逐文件深度解析**（★ 新增）—— 按文件逐个过，每个文件提取 ≥ 2 条"只有读了这个文件才知道"的细节。见工作流第 4 步
7. 给"打算复刻者"的清单（必做 / 推荐 / 反模式）
8. 附录：核心文件速查表 + 魔术数清单 + 攻击向量×防御点对照表（如果是安全相关代码）

参考标杆：本仓库中 [src/tools/BashTool/detail.md](../../../../src/tools/BashTool/detail.md) 就是这种文档的"金标准"。新生成的 detail.md 应该达到这个深度，**并且行数符合上述硬性指标**。

## 工作流

### 第 1 步 —— 摸底 + 建立"文件逐行注释表"

不要直接动手读代码。先用 `ls`/`wc -l` 弄清总行数和文件分布。

```bash
ls -la <folder>/ && echo "---" && wc -l <folder>/*.{ts,tsx,js,py,go,rs} 2>/dev/null
```

做完后**立刻计算目标行数**（根据输出契约的表格），写 scratchpad 第一行：
```
目标：<X> 行。原因：源码共 <Y> 行，每个文件至少 <Z> 段。
```

同时判断文件夹的语言/框架，后面整理细节时按对应生态的关注点来组织。

### 第 2 步 —— 全局骨架（无论大小，必做）

按这个顺序看：

1. **入口文件**：通常是 `<name>.tsx`、`index.ts`、`__init__.py`、`mod.rs`
2. **toolName / constants**：常量定义文件（`toolName.ts:2 行` 这种最小文件往往透露循环依赖等设计）
3. **每个文件的 `^export` 列表**：grep 一下就知道每个文件的"对外能力"
4. **prompt 文件**（如果存在）：把这块代码"如何向 LLM 自我介绍"读完

读这一遍只为回答一个问题：**这堆文件谁是谁的入口、谁调谁、谁纯工具**。

### 第 3 步 —— 全量/深读（按规模决定策略）

#### 策略 A：源码 < 500 行

全量读取所有文件。写完后跳过第 4 步。

#### 策略 B：500 行 ≤ 源码 < 2000 行

1. 全量读取所有文件（Read 全部 .ts/.tsx/.py/.go 文件，大文件用 limit/offset）
2. 读取过程中**每个文件记至少 2 条 scratchpad 笔记**
3. 不满足 2 条的文件必须重读再找

#### 策略 C：2000 行 ≤ 源码 < 5000 行

1. 先 grep `^export` 拿全局骨架
2. **大文件（>200 行）**：至少分段读 40% 的代码量（不能只读开头和 export）
3. **小文件（≤ 200 行）**：全文通读
4. 读取过程中**每个文件记至少 3 条 scratchpad 笔记**
5. 大文件记至少 5 条

#### 策略 D：≥ 5000 行

1. 先 grep `^export`、`SECURITY:`、`#\d{4,}`（issue 引用）、魔术数模式拿全局骨架
2. **大文件（>300 行）**：分段读取不少于 30% 代码量。重点关注：
   - 安全注释（SECURITY、CVE、bypass）
   - 魔术数（数字常量 > 100 的必须有注释说明为什么这么大）
   - 不动点迭代（while 循环 + Set）
   - 入口/出口函数
3. **中等文件（100-300 行）**：至少读 50%
4. **小文件（< 100 行）**：全文通读
5. 每个文件**至少 4 条 scratchpad**，大文件至少 10 条

**关键原则**：跳过文件 = 放弃细节。每个文件无论多小都必须有一个"为什么存在"的解释。如果文件只是一个类型定义或常量导出，就写"为什么这个常量必须单独放在一个文件"（通常是循环依赖）。

### 第 4 步 —— 写"逐文件深度解析"章节（★ 核心新增）

在"关键工程细节"（按模式聚类）之后，新增一节 **"逐文件深度解析"**。这一节**不按模式按文件**，一个一个过每个源文件。

格式：
```
### <filename>

[<filename>:行号](path#L行号) 的 XXX 设计：

**关键细节 1**：[file:line](file#Lline) — 解释
> 引用关键代码 3–8 行（如果需要）

**关键细节 2**：[file:line](file#Lline) — 解释
...

**为什么这个文件存在**：一句话说明这个文件在模块中的角色，以及如果删除它会影响什么。
```

**硬性要求**：
- 每个普通文件 ≥ 2 个关键细节
- 大文件（> 200 行）≥ 4 个关键细节
- 极简文件（< 30 行）可以只写"为什么存在" + 1 个细节
- 本节的**总行数应占整个 detail.md 的 40% 以上**

### 第 5 步 —— 抽取共性模式 + 写 detail.md

读完所有文件并做完 scratchpad 后：

1. 把 scratchpad 里的所有点过一遍 [patterns.md](references/patterns.md) 的 checklist 做归类
2. 按 [output-template.md](references/output-template.md) 的结构写
3. 写作时必须遵守以下**守则**

#### 守则 1：每个细节都要有"反面教材"

不要只写"做了 X"，要写"如果**不**做 X，会发生什么"：

❌ 差：「我们用不动点迭代剥离 wrapper 和 env var」
✅ 好：「单趟剥离会漏：`nohup FOO=bar timeout 5 claude` 第一趟剥 nohup → `FOO=bar timeout 5 claude`，第二趟需要继续；不迭代到不动点，攻击者就能用三层包裹绕过 `Bash(claude:*)` deny」

每一条工程细节理想结构：
```
[现象] → [攻击/失败场景] → [防御/修复] → [代码定位 file:line]
```

#### 守则 2：保留作者注释中的"血泪"

作者花时间写在注释里的 issue 引用、HackerOne 编号、攻击 trace —— **必须原样搬运**到 detail.md 里。复刻者最需要这些。

#### 守则 3：所有代码定位都用 markdown 链接

格式：`[file:line](path#Lline)` 或 `[file:line-line2](path#Lline-Lline2)`

便于读者 cmd+click 跳转源码核对。**禁止**用反引号包文件名 —— 不可点击。

#### 守则 4：魔术数必须解释

任何代码里的具体数字（超时、阈值、buffer size、重试次数）都要附上"为什么是这个值"。如果作者注释没说，自己读上下文推断（找它被用在哪里，估算约束）。如果没有上下文可以推断，写"作者未注明，从上下文推测是 X"，**不要**编一个理由。

#### 守则 5：把"反模式清单"写出来

最后必有一节"反模式（不要这样做）"：列出代码作者**显然刻意避开**的写法，并解释代价。这一节给复刻者最大价值。

#### 守则 6：示例代码要短而完整

引用源码代码块的时候，**贴 5–15 行最关键的部分**，前后用注释说"省略了 X"。不要贴 50 行不带解释的代码。

#### 守则 7：必须覆盖每个文件的魔术数

每看到一个常量数字 > 100 或超时 > 1 秒，就要问"为什么是这个值"。在附录的魔术数清单中汇总。

#### 守则 8：行数达标硬指标

写完整篇 detail.md 后必须用 `wc -l` 检查行数是否达标。如果低于下限，说明细节不够，必须补充"逐文件深度解析"节的每个文件条目。

### 第 6 步 —— 自检

写完后用 [self-check.md](references/self-check.md) 的清单过一遍。任何一项 No 都要补：

- [ ] 文件分工表覆盖了**所有**源码文件？
- [ ] 每个文件至少在 detail 中被提到 ≥ 1 次（速查表 + 逐文件深度解析节）
- [ ] 大文件提及次数 ≥ 要求次数（见输出契约的表）
- [ ] 至少 5 条"攻击 → 防御"对照？（如果是安全相关代码）
- [ ] 所有魔术数都解释了？
- [ ] 引用代码都用 `[file:line](path#Lline)` 链接？
- [ ] 有"反模式（不要做）"清单？
- [ ] 有给复刻者的 minimal checklist？
- [ ] 输出语言一致（中文讲解 + 英文代码/术语）？
- [ ] 与原代码注释里写明的 issue/PR/CVE 全部捕获？
- [ ] **`wc -l detail.md` 的结果 ≥ 输出契约对应行数下限**？

### 第 7 步 —— 写入文件

把 detail.md 写到 `<入参文件夹>/detail.md`。如果该文件已存在，直接覆盖（用户重新触发说明想刷新内容）。

写完后告诉用户：
- 文件位置（用 markdown 链接）
- 总行数 + 源码总行数 + 覆盖率（detail.md行数 / 源码行数 × 100%）
- 覆盖了多少个文件，"逐文件深度解析"节有几段
- 1–2 句话点出最有价值的洞察（吸引用户去读）

## 关键提醒

- **本 skill 的核心是 detail，不是概览**。如果用户只想要快速概览，引导他用 `/init` 或别的工具，**不要**勉强生成又长又水的 detail。
- **不要发明细节**。所有"为什么这么写"必须有源码注释或代码逻辑支撑。如果作者没解释一个魔术数，写"作者未注明，从上下文推测是 X"，**不要**编一个理由。
- **不要重复源码**。整文照搬代码毫无价值。永远用"现象 + 解释 + 链接"格式。
- **行数与源码规模成正比**。这是本版更新的核心修改：1800 行源码 → 600 行 detail.md，12000 行源码 → 2500+ 行 detail.md。行数不够 = 细节没挖透。但也不要为凑行数注水。
- **"逐文件深度解析"节不是可选的**。它是保证每个文件都被覆盖到的强制措施。如果某个文件你写不出 2 个关键细节，说明你没读懂，回去重读。

# Replication Guide — 可复刻工程实现细节生成器

## 这个 skill 解决什么

用户经常想"借鉴一下某个开源项目某块代码的实现"。但直接读源码代价高：
- 注释里藏着大量"为什么这么写"的故事，但散落在几十个文件里
- 攻击向量、bug 修复、不对称设计这些**最有价值的细节**正是要全文通读才能拼齐的
- 普通的"代码概览"文档抓不住这些 —— 它们关心架构、不关心血泪

本 skill 把这件事系统化：给一个文件夹路径，输出一份 **detail.md**，让用户能**只看这份文档就把功能等价的代码重新写一遍**。

## 输出契约

文件路径：`<入参文件夹>/detail.md`（直接放在被分析的文件夹里）

文件内容：约 500–1500 行 Markdown，结构遵循 [output-template.md](references/output-template.md)，至少覆盖：

1. 整体哲学（这块代码假设什么、解决什么）
2. 文件分工速查表（每个文件多少行、做什么）
3. 接口/门面（对外暴露的形状）
4. 核心数据流（一次典型调用从入口到出口的完整路径）
5. **关键工程细节** —— 这一节是核心，越多越好：
   - 多层防御 / 多模式校验
   - 不动点迭代 / 顺序敏感操作
   - 模式之间的不对称（allow vs deny / 模型视角 vs 用户视角）
   - 解析失败时的 fail-safe 方向
   - 注释里写明的攻击向量与对应防御
   - 一字之差的安全细节（如 `\s+` vs `[ \t]+`）
6. 给"打算复刻者"的清单（必做 / 推荐 / 反模式）
7. 附录：核心文件速查表 + 攻击向量×防御点对照表（如果是安全相关代码）

参考标杆：本仓库中 [src/tools/BashTool/detail.md](../../../../src/tools/BashTool/detail.md) 就是这种文档的"金标准"。新生成的 detail.md 应该达到这个深度。

## 工作流

### 第 1 步 —— 摸底

不要直接动手读代码。先用 `ls`/`wc -l` 弄清两个数字：

| 数字 | 决策 |
|------|------|
| 总行数 < 1500 | **全量读取**：所有文件一口气读完再写 |
| 1500 ≤ 总行数 < 5000 | **全量读取 + 优先级**：每个文件至少扫一遍，关键文件细读 |
| 总行数 ≥ 5000 | **多层抽取**：先 grep `^export`/`^class`/`^function` 拿全局骨架，再按重要性分组细读 |

```bash
ls -la <folder>/ && echo "---" && wc -l <folder>/*.{ts,tsx,js,py,go,rs} 2>/dev/null
```

> 这一步顺便要判断文件夹的语言/框架（TypeScript？Python？React+TUI？），后面整理细节时按对应生态的关注点来组织。

### 第 2 步 —— 全局骨架（无论大小，必做）

按这个顺序看：

1. **入口文件**：通常是 `<name>.tsx`、`index.ts`、`__init__.py`、`mod.rs`
2. **toolName / constants**：常量定义文件（`toolName.ts:2 行` 这种最小文件往往透露循环依赖等设计）
3. **每个文件的 `^export` 列表**：grep 一下就知道每个文件的"对外能力"
4. **prompt 文件**（如果存在）：把这块代码"如何向 LLM 自我介绍"读完

读这一遍只为回答一个问题：**这堆文件谁是谁的入口、谁调谁、谁纯工具**。

### 第 3 步 —— 深读关键模块（核心环节）

按这个**优先级**深读，每读一段都问自己**5 个问题**（见下文）：

**优先级排序**：
1. 主入口（buildTool / class XXX / def main）—— 看出整体形状
2. 安全/校验/权限 —— **这里的注释含金量最高**，作者经常写明攻击场景
3. 数据流转换函数 —— 输入 → 输出之间的 N 步处理
4. 工具函数 / utils —— 常常藏着"一字之差"的细节
5. UI / 渲染 —— 对复刻者较次要但要点出"双视图分离"这种模式

**5 个永远要问的问题**（这是本 skill 的灵魂）：

| # | 问题 | 找什么线索 |
|---|------|-----------|
| 1 | 为什么这里**多了一个看似冗余的步骤**？ | "// SECURITY:"、"// Defense in depth"、"_DEPRECATED" 后缀、注释里写"belt and suspenders" |
| 2 | 这段代码**修过哪个 bug**？ | issue 编号、PR 编号、HackerOne 引用、"// Previously: ..."、"// Fix for #..." |
| 3 | 这两个看起来一样的逻辑**为什么不对称**？ | allow vs deny、模型 vs 用户、严格 vs 宽松；注释里出现"asymmetric"/"intentional" |
| 4 | 解析/校验**失败了走哪条路**？ | "fail safe"、"fail closed"、"fail in the safe direction"、`return () => true` 这种 fallback |
| 5 | 这个常量为什么是这个**具体的值**？ | 30000、15_000、64*1024*1024、2 秒等魔术数 → 注释里通常写明阈值由来 |

**每发现一个值得提的细节，立刻在 scratchpad 里记一行**：
```
[文件:行号] 类型(攻击/不动点/不对称/魔术数/fail-safe) 一句话总结
```

### 第 4 步 —— 抽取共性模式

读完后，把 scratchpad 里的所有点过一遍 [patterns.md](references/patterns.md) 的 checklist 做归类。常见的可归类模式：

- **多层防御**：同一个攻击向量在 ≥ 2 处被堵
- **不动点迭代**：循环到无新候选才停
- **AST 优先 regex 兜底**
- **lazy 初始化破循环依赖**
- **白名单而不是黑名单**
- **静默失败 = 安全 bug**（开关失效必须告警）
- **模型视图 ≠ 用户视图**
- **顺序敏感**：A 必须在 B 之前/之后
- **零 token 侧信道**
- **可中断 / 可后台 / 可恢复**

如果发现 ≥ 3 个模式，[output-template.md](references/output-template.md) 里"关键工程细节"那节就按模式聚类来写，不要按文件聚类。

### 第 5 步 —— 写 detail.md

按 [output-template.md](references/output-template.md) 的结构写。写作时遵守下面这几条 **写作守则**：

#### 守则 1：每个细节都要有"反面教材"

不要只写"做了 X"，要写"如果**不**做 X，会发生什么"：

❌ 差：「我们用不动点迭代剥离 wrapper 和 env var」
✅ 好：「单趟剥离会漏：`nohup FOO=bar timeout 5 claude` 第一趟剥 nohup → `FOO=bar timeout 5 claude`，第二趟需要继续；不迭代到不动点，攻击者就能用三层包裹绕过 `Bash(claude:*)` deny」

每一条工程细节理想结构：
```
[现象] → [攻击/失败场景] → [防御/修复] → [代码定位 file:line]
```

#### 守则 2：保留作者注释中的"血泪"

作者花时间写在注释里的 issue 引用、HackerOne 编号、攻击 trace —— **必须原样搬运**到 detail.md 里。复刻者最需要这些。

#### 守则 3：所有代码定位都用 markdown 链接

格式：`[file:line](path#Lline)` 或 `[file:line-line2](path#Lline-Lline2)`

便于读者 cmd+click 跳转源码核对。**禁止**用反引号包文件名 —— 不可点击。

#### 守则 4：魔术数必须解释

任何代码里的具体数字（超时、阈值、buffer size、重试次数）都要附上"为什么是这个值"。如果作者注释没说，自己读上下文推断（找它被用在哪里，估算约束）。

#### 守则 5：把"反模式清单"写出来

最后必有一节"反模式（不要这样做）"：列出代码作者**显然刻意避开**的写法，并解释代价。这一节给复刻者最大价值。

#### 守则 6：示例代码要短而完整

引用源码代码块的时候，**贴 5–15 行最关键的部分**，前后用注释说"省略了 X"。不要贴 50 行不带解释的代码。

### 第 6 步 —— 自检

写完后用 [self-check.md](references/self-check.md) 的清单过一遍。任何一项 No 都要补：

- [ ] 文件分工表覆盖了所有文件？
- [ ] 至少 5 条"攻击 → 防御"对照？（如果是安全相关代码）
- [ ] 所有魔术数都解释了？
- [ ] 引用代码都用 `[file:line](path#Lline)` 链接？
- [ ] 有"反模式（不要做）"清单？
- [ ] 有给复刻者的 minimal checklist？
- [ ] 输出语言一致（中文讲解 + 英文代码/术语）？
- [ ] 与原代码注释里写明的 issue/PR/CVE 全部捕获？

### 第 7 步 —— 写入文件

把 detail.md 写到 `<入参文件夹>/detail.md`。如果该文件已存在，直接覆盖（用户重新触发说明想刷新内容）。

写完后告诉用户：
- 文件位置（用 markdown 链接）
- 章节数 + 大致字数
- 1–2 句话点出最有价值的洞察（吸引用户去读）

## 关键提醒

- **本 skill 的核心是 detail，不是概览**。如果用户只想要快速概览，引导他用 `/init` 或别的工具，**不要**勉强生成又长又水的 detail。
- **不要发明细节**。所有"为什么这么写"必须有源码注释或代码逻辑支撑。如果作者没解释一个魔术数，写"作者未注明，从上下文推测是 X"，**不要**编一个理由。
- **不要重复源码**。整文照搬代码毫无价值。永远用"现象 + 解释 + 链接"格式。
- **行数不是越多越好**。如果文件夹小，detail.md 也应该小。3000 行的 detail 写一个 200 行的工具是失败 —— 说明在掺水。

# detail.md 写完后的自检清单

写完后逐项过一遍。任何一项 No 都要补。**作弊不写自检** = 用户拿到一份会让他白跑一趟的文档。

## 完整性（★ 更新：强调逐文件覆盖）

- [ ] 文件分工表覆盖了文件夹里**所有**源码文件（测试文件可以省略）
- [ ] **"逐文件深度解析"节覆盖了文件夹里每个非空源码文件**（每个文件都有独立小节）
- [ ] 每个文件至少有最小的细节量（见输出契约的"每个源文件的最小出现次数"表）
- [ ] 对外接口章节列出了**所有公开导出**
- [ ] 核心数据流图覆盖从入口到出口
- [ ] 每个工程模式至少 1 个具体例子（不是抽象描述）

## "为什么"含金量

- [ ] 至少 8 处 **"现象 → 攻击/失败场景 → 防御/修复 → 代码定位"** 完整四元组（中型文件夹以上至少 15 处）
- [ ] 所有魔术数（出现的具体数字）都有解释
- [ ] 作者注释里的 **issue 编号 / PR 编号 / HackerOne 引用 / CVE** 全部捕获
- [ ] 至少 2 处不对称设计（A 严格 / B 宽松）的 why
- [ ] 至少 2 处"如果不这么写会怎样"的反面教材（最好每节都有）

## 防止掺水

- [ ] 没有"这个文件实现了 xxx"这种空话（必须说**具体**实现了什么 + 怎么实现）
- [ ] 没有大段照抄源码而不解释
- [ ] 单个代码块 ≤ 15 行，超过的用 `// ... 省略` 缩短
- [ ] 没有"显然"/"明显"/"简单地"这种把读者赶走的措辞
- [ ] **行数符合更新后的规模矩阵（硬性指标）**：
  - 源码 < 500 行 → detail.md ≤ 300 行
  - 源码 500–2000 行 → detail.md 500–800 行
  - 源码 2000–5000 行 → detail.md 800–1500 行
  - 源码 5000–10000 行 → detail.md 1500–2500 行
  - 源码 > 10000 行 → detail.md 2500+ 行（上不封顶）

## 可点击性

- [ ] **所有**文件引用都用 markdown 链接 `[file:line](path#Lline)`，不是反引号包裹
- [ ] 链接路径相对于 detail.md 所在目录（detail.md 在文件夹里 → `[xxx.ts:100](xxx.ts#L100)`）
- [ ] 章节内引用用锚点链接

## 复刻可用性

- [ ] 有"反模式（不要做）"清单（≥ 5 条，中型文件夹以上 ≥ 8 条）
- [ ] 有"必做 / 推荐"分级清单
- [ ] 一个不熟悉源代码的工程师，**只读 detail.md** 能写出功能等价的代码？
  - 接口形状清楚？
  - 安全检查列表清楚？
  - 关键算法（不动点、AST 处理）描述够细？
  - 边界情况列出来了？
  - 每个文件的角色和设计意图都清楚了？（"逐文件深度解析"节提供了这个）

## 输出语言

- [ ] 中文讲解（除非用户指定别的语言）
- [ ] 代码、术语、注释、错误信息**保留英文**（翻译反而失真）
- [ ] 标点符号和空格规范（中英文之间空格）

## 一句话总结

- [ ] 文末"一句话总结"做到了**用一句犀利的话点出整个模块的核心取舍**
- [ ] 这一句话单拿出来 tweet 也能站得住

- [ ] 我引用了一个**不存在**的文件/行号吗？
- [ ] 我把一个**不在**这个文件夹的代码当成"在文件夹里"写进 detail 了吗？

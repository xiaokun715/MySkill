# detail.md 写完后的自检清单

写完后逐项过一遍。任何一项 No 都要补。**作弊不写自检** = 用户拿到一份会让他白跑一趟的文档。

## 完整性

- [ ] 文件分工表覆盖了文件夹里**所有**源码文件（测试文件可以省略）
- [ ] 每个文件至少在 detail 中被提到一次（速查表 + 至少一节正文）
- [ ] 对外接口章节列出了**所有公开导出**
- [ ] 核心数据流图覆盖从入口到出口
- [ ] 每个工程模式至少 1 个具体例子（不是抽象描述）

## "为什么"含金量

- [ ] 至少 5 处 **"现象 → 攻击/失败场景 → 防御/修复 → 代码定位"** 完整四元组
- [ ] 所有魔术数（出现的具体数字）都有解释
- [ ] 作者注释里的 **issue 编号 / PR 编号 / HackerOne 引用 / CVE** 全部捕获
- [ ] 至少 1 处"如果不这么写会怎样"的反面教材（最好每节都有）
- [ ] 至少 1 处不对称设计（A 严格 / B 宽松）的 why

## 防止掺水

- [ ] 没有"这个文件实现了 xxx"这种空话（必须说**具体**实现了什么 + 怎么实现）
- [ ] 没有大段照抄源码而不解释
- [ ] 单个代码块 ≤ 15 行，超过的用 `// ... 省略` 缩短
- [ ] 没有"显然"/"明显"/"简单地"这种把读者赶走的措辞
- [ ] 行数符合规模：
  - 源码 < 1500 行 → detail.md ≤ 500 行
  - 源码 1500–5000 行 → detail.md 500–1000 行
  - 源码 > 5000 行 → detail.md 1000–2000 行

## 可点击性

- [ ] **所有**文件引用都用 markdown 链接 `[file:line](path#Lline)`，不是反引号包裹
- [ ] 链接路径相对于 detail.md 所在目录（detail.md 在文件夹里 → `[xxx.ts:100](xxx.ts#L100)`）
- [ ] 章节内引用用锚点链接

## 复刻可用性

- [ ] 有"反模式（不要做）"清单（≥ 5 条）
- [ ] 有"必做 / 推荐"分级清单
- [ ] 一个不熟悉源代码的工程师，**只读 detail.md** 能写出功能等价的代码？
  - 接口形状清楚？
  - 安全检查列表清楚？
  - 关键算法（不动点、AST 处理）描述够细？
  - 边界情况列出来了？

## 输出语言

- [ ] 中文讲解（除非用户指定别的语言）
- [ ] 代码、术语、注释、错误信息**保留英文**（翻译反而失真）
- [ ] 标点符号和空格规范（中英文之间空格）

## 一句话总结

- [ ] 文末"一句话总结"做到了**用一句犀利的话点出整个模块的核心取舍**
- [ ] 这一句话单拿出来 tweet 也能站得住

---

## 红线检查（任何一条 Yes 都不能交付）

- [ ] 我编了某个细节但没有源码依据吗？（"为什么是 30000"如果作者没说，我自己编了一个理由？）
- [ ] 我引用了一个**不存在**的文件/行号吗？
- [ ] 我把一个**不在**这个文件夹的代码当成"在文件夹里"写进 detail 了吗？
- [ ] 我跳过了某个关键文件因为"没时间读"了吗？
- [ ] 我用 detail.md 替换掉了一份**已经存在的**好文档吗？（应该问用户：是否真的覆盖）

## 通过标准

通过 ≥ 95% 完整性、≥ 90% 含金量、100% 红线 → 交付。

否则回去补 / 重写。

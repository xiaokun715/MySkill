# 不同规模文件夹的读取策略（更新版）

写 detail.md 之前，先用 `wc -l` 估算总行数，按下面策略走。**核心变化**：每个文件都必须覆盖到，"逐文件深度解析"节强制保证覆盖率。

## 摸底命令模板

```bash
# 1. 文件分布
ls -la <folder>/

# 2. 行数与语言
wc -l <folder>/*.{ts,tsx,js,py,go,rs,java,cpp,h} 2>/dev/null | sort -n

# 3. 全局骨架（每个文件的 export）
grep -n "^export\|^class\|^function\|^async function\|^const" <folder>/*.{ts,tsx,js,py,go,rs} 2>/dev/null | head -100

# 4. 安全相关注释（含金量最高的注释）
grep -rn "SECURITY:\|CVE-\|HackerOne\|Defense in depth\|fail.safe\|fail.closed\|fail.open\|attacker\|bypass\|TODO.*security\|FIXME" <folder>/ 2>/dev/null | head -50

# 5. issue / PR 引用（修复痕迹）
grep -rn "#[0-9]\{4,\}\|PR #\|issue.*[0-9]\{4,\}" <folder>/ 2>/dev/null | head -30

# 6. 魔术数（数字常量，往往透露设计思路）
grep -rn "= [0-9]\{4,\}\|_MS = \|_BYTES = \|_LIMIT = \|_MAX_\|_TIMEOUT\|_THRESHOLD\|_BUDGET" <folder>/ 2>/dev/null | head -40
```

## 策略 A：< 500 总行数（极小型文件夹）

**做法**：
1. 一次性 Read 所有文件
2. 每个文件至少 1 个实现细节
3. 不需要"逐文件深度解析"独立成节，合并到关键工程细节中
4. 输出 detail.md ≤ 300 行

**适用例子**：单个小工具、常量集合、几个 utility 函数。

## 策略 B：500 – 2000 总行数（小型文件夹）

**做法**：
1. 一次性 Read 所有文件。对 > 300 行的文件用 limit/offset 分段读
2. 每个文件在 scratchpad 中记至少 2 条笔记
3. 写 detail 时：**关键工程细节（按模式聚类）** + **逐文件深度解析**两节并存
4. "逐文件深度解析"节中每个文件至少 2 个关键细节
5. 输出 detail.md 500–800 行

**适用例子**：单个工具（FileEditTool、GlobTool 等）。

## 策略 C：2000 – 5000 总行数（中型文件夹）

**做法**：
1. 用摸底命令的 #3 拿全局骨架，识别"主入口 + 核心 + 工具"三类文件
2. **大文件（> 200 行）**：分段读不少于 40% 的代码
3. **中等文件（50-200 行）**：全文通读
4. **小文件（< 50 行）**：grep export + 扫一眼注释
5. scratchpad 中每个文件至少 3 条笔记，大文件至少 5 条
6. 写 detail 时："关键工程细节（按模式聚类）" + "逐文件深度解析"两节
7. "逐文件深度解析"节中每个小文件 ≥ 2 段，大文件 ≥ 5 段
8. 输出 detail.md 800–1500 行

**适用例子**：完整的工具模块（AgentTool、FileReadTool 等）、小型子系统。

## 策略 D：5000 – 10000 总行数（大型文件夹）

**做法**：
1. **必须先 grep 骨架**，禁止盲读所有文件
2. 大文件（> 300 行）：分段读不少于 30% 代码，重点关注安全注释 + 魔术数 + 不动点迭代 + 入口/出口函数
3. 中等文件（100-300 行）：至少读 50%
4. 小文件（< 100 行）：全文通读
5. **每个文件至少 4 条 scratchpad**，大文件至少 8 条
6. 写 detail 时："关键工程细节（按模式聚类）" + "逐文件深度解析"两节严格并存
7. "逐文件深度解析"节中每个文件至少 4 段，大文件至少 8 段
8. 可以有多个"逐文件"子节（如"核心层"、"工具层"分组）
9. 输出 detail.md 1500–2500 行

**适用例子**：BashTool（12,400 行）、完整的 MCP 服务层。

## 策略 E：≥ 10000 总行数（超大型文件夹）

**做法**：
1. 先骨架扫描，按"重要性 × 复杂度"给文件打分，分 3 组：核心（> 50%）/ 重要（30%）/ 辅助（20%）
2. 核心组文件：测读不少于 50% 代码量
3. 重要组文件：读不少于 30%
4. 辅助组文件：通读或扫 export
5. **每个文件至少 5 条 scratchpad**，核心文件至少 12 条
6. "逐文件深度解析"节中每个文件至少 5 段，核心文件至少 10 段
7. 输出 detail.md 2500+ 行，上不封顶
8. **必须**有附录：核心文件速查表 + 模式 × 文件交叉表 + 魔术数完整清单

**适用例子**：完整的 Bash 工具栈（含 sandbox + permissions + AST 解析）、整个 Agent 调度系统。

### 超大型文件夹的"层进式"展开

```
第 1 轮（5 分钟）：grep + ls + wc，写完文件分工表
第 2 轮（15 分钟）：读所有 ≤ 100 行的文件全文 + 大文件的 export 列表
第 3 轮（30 分钟）：定位关键函数（SECURITY 注释、issue 引用、不动点循环）
第 4 轮（40 分钟）：逐文件深读核心组，每文件记 ≥ 5 条 scratchpad
第 5 轮（40 分钟）：写 detail.md（模式聚类节 + 逐文件深度解析节）
第 6 轮（15 分钟）：自检 + 补读"写不出 2 段的文件"
```

## 单文件 > 2000 行的处理

无论文件夹规模，遇到 > 2000 行的单文件：

```bash
# 先看结构
grep -n "^export\|^const\|^function\|^class\|^async function" <file> | head -40

# 再按段读
Read <file> --offset N --limit 200  # 一次最多读 200 行
```

对于这种超大文件，在"逐文件深度解析"节中可以拆成 2-3 个子节（如"连接管理"、"认证流程"、"结果处理"），保证该文件的细节篇数达标。

## 常见问题

**问**：如果一个文件的代码太简单（< 30 行纯类型定义），写不出 2 个细节怎么办？
**答**：可以只写"为什么这个文件存在"（通常是循环依赖或类型隔离），加上一个关于 schema 设计的细节。但必须提到。

**问**：如何确保每个文件都写到了？
**答**：写作时拿文件列表逐行检查。写完"逐文件深度解析"节后，对照 `ls -la <folder>/` 的结果逐一核对。

**问**：超过 5000 行的文件夹，写 2500 行 detail.md 是不是太长了？
**答**：不是。2500 行对于 5000 行源码来说覆盖率只有 50%。BashTool 有 12,400 行源码，BashTool 的 detail.md 应达到 2500-4000 行。源码细节密度决定了 detail.md 的长度。

## 不要做的事

❌ **盲目 head 第一个文件就开写** —— 没读完就动笔会缺乏全局视角
❌ **大文件夹不做"逐文件深度解析"节** —— 这是更新版的强制要求
❌ **小文件夹做策略 D/E** —— 短代码硬凑长文档会掺水
❌ **不做摸底命令直接 Read** —— 对大文件夹一头雾水
❌ **跳过安全注释 grep** —— 错过最有价值的"why"线索
❌ **写完整篇不 wc -l 检查行数** —— 行数不达标 → 细节不够
